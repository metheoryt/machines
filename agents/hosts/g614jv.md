## Native Windows

### Claude config bootstrap — run via Git Bash, NOT PowerShell
- Claude Code runs on win32 and reads `C:\Users\methe\.claude`. To (re)link the
  tracked config there, run the bootstrap through **Git Bash** (`HOME=C:\Users\methe`):
  `& "C:\Program Files\Git\bin\bash.exe" agents/bootstrap.sh`.
- Do NOT run `bash agents/bootstrap.sh` from PowerShell — that `bash` resolves to
  **WSL's**, whose `HOME=/home/me`, so it links the repo into the WSL home instead
  of the Windows `.claude` that Claude Code actually uses. Symptom: a newly-added
  hooks/skills/agents entry shows up missing on Windows (e.g. the SessionStart
  `gortex-onboard-check.sh` "No such file or directory" error) even though a WSL
  bootstrap reported success.

### Port 22 is answered by Windows sshd, not WSL
On this box, port 22 may be answered entirely by Windows' own OpenSSH server
(banner `OpenSSH_for_Windows_9.5`), not the WSL distro's sshd — a key in the
WSL distro's `~/.ssh/authorized_keys` has zero effect if only Windows sshd
listens. For the admin Windows account the trusted-keys file is
`C:\ProgramData\ssh\administrators_authorized_keys` (ACL: SYSTEM +
Administrators), not per-user `~/.ssh/authorized_keys`.

### GitHub SSH identities (two accounts)
- Windows `~/.ssh/id_rsa` = **metheoryt** (default `git@github.com`); `~/.ssh/id_ed25519` = **cyphy671**, used via the `githubcyphy` host alias in `~/.ssh/config`. Verify with `ssh -T git@github.com` / `ssh -T git@githubcyphy`.
- `cyphy671` owns the **qaz-code** repo (`git@githubcyphy:cyphy671/qaz-code.git`; local Windows dir is `GitHub\qaz-law`). `metheoryt` owns machines, airdrome, vasya, etc.

### Apps & tooling quirks
- **Google Drive's streaming mount letter is auto-assigned, not pinned** — unplugging
  another removable drive (USB / backup SSD) can shift it to a different letter on the
  next boot and break any app config that hardcodes the old letter (e.g. an Obsidian
  vault registered at `I:\Мой диск\…`). Pin it under Google Drive settings.
- **AyuGram** (the Telegram fork installed here) is a *portable* winget package whose
  `tdata` session folder lives next to the `.exe` in the winget package dir, NOT under
  `%APPDATA%` — a winget update can wipe it, and two `tdata` folders can't be merged
  (replace wholesale). Back it up before updating.

### Tooling gotcha — PowerShell tool + `wsl … bash -c`
- Invoking `wsl -d <distro> -u me -- bash -c '…$var…'` **through the PowerShell tool loses shell variables** (they expand empty) and cwd falls back to the `/mnt/c` Windows dir. Fix: write the script to a file with LF endings (use the Bash/Git-Bash tool) and run it via `wsl -d <distro> -u me -- bash /mnt/c/…/script.sh`.

### Split toolset between Git Bash and WSL
- Windows Git Bash has `just` + `pwsh` but **no `jq`**; the default WSL distro
  (Ubuntu-26.04) has `jq` but **no `just`** — so verifying a `just`+`jq` pipeline
  (e.g. `just provision`) means running the recipe's shell body inside WSL, and the
  literal `just provision` invocation is deferred to a real nixos/`jq`+`just` box.
- **Root-owned `__pycache__/*.pyc`** left by a Docker container that ran as root can
  block a plain `rm -rf` of a repo dir on this WSL box, since the WSL `me` user has no
  passwordless sudo — clear them with an interactively-run `sudo rm -rf`.
